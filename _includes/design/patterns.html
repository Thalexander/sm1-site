Design Patterns

We have used design patterns extensively in our development process, these aid maintainability of our code as they can be recognised by other developers and avoids us having to 'reinvent the wheel' by using an accepted solution to a problem.

async-waterfall

One of the common, recurring patterns in the bot code are asynchronous function handlers. As Javascript is an asynchronous language, callbacks often have to be used to take an action in response to a result. This can lead to a lot of nesting! The async-waterfall pattern keeps code tidy and reduces boilerplate by keeping functions to be executed sequentially defined in a flat array, passing the results of one function to the next in the array when the previous function completes.

Dependency Injection

Dependency injection refers to the ability to provide a dependency that an object needs, without requiring that object to construct it itself. The intent is to decouple specific implementations from client code and allow dependencies to be swapped out easily. We have used dependency injection to quickly programmatically test our chatbot without having to deploy it as a web service every time. In this case, dependency injection has allowed us to swap out the web connector with a console connector, enabling us to test the bot in a local console during development instead of having to deploy it to a full blown webserver, improving our iteration times.

Method Chaining

Method chaining involves functions returning the current object itself so that function calls can be chained together without requiring any intermediate variables. When appropriate, we made use of method chaining to make our code more concise.

Chain of Responsibility

The chain-of-responsibility pattern refers to having a group of processing objects that handle specific actions and passing actions along this chain until one of the processing objects handle it, at which point it is consumed. This gives more than one object an opportunity to handle the action. In our chatbot, our dialogue handlers are structured in this way, with dialogue intents chained in order of priority. This allowed us to match and route a wide variety of intents to the right handlers, including bot commands (e.g. /login) as well as intents extracted from natural language processing. (e.g. Find Item)
